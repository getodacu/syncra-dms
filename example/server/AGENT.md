# Server Agent Conventions

These rules apply inside `server/` in addition to the root conventions.

## Stack

- Go module: `ai.ro/syncra`.
- HTTP framework: Gin.
- Database: PostgreSQL via GORM and pgx-related dependencies.
- Configuration: environment variables loaded through `server/.env`.
- API docs: Swagger JSON generated by `go run ./cmd/syncra swagger`.
- OCR work is split across API handlers, domain services in `internal/ocr`, executor code, and Atlas database migrations.

## Database Rules

- Development must use the `syncra_dev` database.
- In worktrees, copy `server/.env` from the source checkout before running server commands, then verify its `DSN` targets `syncra_dev`.
- `server/.env` must have a `DSN` whose `dbname` is `syncra_dev` before running the API, migrations, executor flows, or manual database experiments.
- `server/.env` must have a `DSN_DEV` whose `dbname` is `syncra_dev` before running Postgres-backed tests; tests fail fast rather than falling back to `DSN`.
- `server/.env` must also have `ATLAS_DATABASE_URL` pointing at the same `syncra_dev` database in URL format, for example `postgres://user:password@localhost:5432/syncra_dev?search_path=public&sslmode=disable`.
- `server/.env` must have `ATLAS_DEV_DATABASE_URL` pointing at an empty scratch database for Atlas planning and linting; do not point it at `syncra_dev`.
- Never run `go run ./cmd/syncra migrate` unless both `DSN` and `ATLAS_DATABASE_URL` have been checked and point at `syncra_dev`.
- Postgres-backed tests use `internal/testsupport`, load `server/.env`, and connect with `DSN_DEV`; keep those tests isolated with grouped transactions or the existing transaction/advisory-lock helpers.
- Atlas owns versioned schema migrations. Do not reintroduce runtime GORM `AutoMigrate` or `database.Migrate` for normal schema changes.
- Keep `internal/dbmigrate` for legacy repair helpers and narrowly scoped data migrations that need Go logic; do not use it as the normal schema migration path.

## Atlas Migrations

- The desired schema comes from GORM models through `cmd/atlas-loader` and `atlas.hcl`.
- The versioned migration directory is `server/migrations/`; keep `migrations/atlas.sum` committed with migration SQL changes.
- Atlas uses `ATLAS_DEV_DATABASE_URL` from the environment as its dev database, so migration generation and linting require the `atlas` CLI and a reachable empty scratch database.
- Before running direct `atlas ... --env local` commands, export `server/.env` into the shell:

```sh
set -a; source .env; set +a
```

- To inspect the desired schema generated from GORM models:

```sh
atlas schema inspect --env local --url env://src --format '{{ sql . }}'
```

- To generate a migration after changing GORM model tags, relationships, indexes, or table names:

```sh
atlas migrate diff <descriptive_name> --env local
```

- Review generated SQL before committing. Preserve intentional manual objects from the baseline, including `pg_trgm`, `idx_ocr_documents_original_filename_trgm`, `fk_ocr_documents_job`, and `chk_ocr_jobs_status`.
- If manually editing, adding, removing, or renaming migration files, refresh the checksum:

```sh
atlas migrate hash --dir file://migrations
```

- Validate migration files and checksums before reporting migration work complete:

```sh
atlas migrate validate --dir file://migrations
atlas migrate lint --env local --latest 1
```

- Apply pending migrations only after confirming `ATLAS_DATABASE_URL` targets `syncra_dev`:

```sh
go run ./cmd/syncra migrate
```

- Check database migration status with:

```sh
atlas migrate status --env local --url "$ATLAS_DATABASE_URL"
```

## Commands

```sh
go test ./...
go test ./internal/api
go test ./internal/ocr
go test ./internal/database ./internal/dbmigrate
go run ./cmd/syncra api
go run ./cmd/syncra migrate
go run ./cmd/syncra swagger
atlas migrate validate --dir file://migrations
atlas migrate lint --env local --latest 1
```

Run commands from `server/`.

## Code Conventions

- Keep HTTP concerns in `internal/api`; keep business/domain behavior in `internal/ocr`, `internal/auth`, or another focused package.
- Keep application startup and dependency wiring in `internal/app` or `cmd/*`, not in handlers.
- Keep database opening and model-list wiring in `internal/database`; keep CLI migration orchestration in `cmd/syncra`.
- Use typed request/response structs and existing response helpers.
- Preserve authentication and ownership checks for user-owned rows.
- Validate inputs at API boundaries and keep provider-specific OCR behavior behind service/executor boundaries.
- Close database handles in long-lived commands and tests when opening non-transactional handles.

## Tests

- For handler changes, add or update tests in `internal/api`.
- For domain behavior, add or update tests next to the relevant `internal/ocr`, `internal/auth`, or `internal/database` code.
- For Atlas schema migrations, validate generated SQL with Atlas and add Go tests for model behavior or API/domain behavior affected by the schema.
- For Go-based legacy repair/data migrations in `internal/dbmigrate`, include tests that cover pre-existing schema states and repeated migration runs.
- Avoid tests that depend on row ordering unless the query explicitly orders rows.
- Keep test fixtures minimal and create only the models needed for the behavior under test.

## Swagger

- Update Swagger annotations/docs when API request/response shapes or routes change.
- After API surface changes, run:

```sh
go run ./cmd/syncra swagger
```

- If the `swagger` binary is missing, report that clearly instead of hand-editing generated Swagger output.
